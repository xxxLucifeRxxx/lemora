# Lemora

**Lemora** — мобильное Flutter-приложение для изучения языков через карточки, интервальные повторения и AI-помощника.

Идея проекта: помочь пользователю не просто запоминать отдельные слова, а постепенно переводить их в активный словарный запас через контекст, примеры, произношение и регулярные повторения.

## Цели проекта

- Создать качественный pet-проект, который можно показывать как пример Flutter-кода.
- Реализовать полезное приложение с понятной продуктовой ценностью.
- Построить архитектуру, которую можно развивать без хаотичного роста кода.
- Применять принципы SOLID, разделение ответственности и тестируемую бизнес-логику.
- Заложить основу для будущей монетизации через Pro-функции.

## Основные сценарии

- Выбор изучаемого языка.
- Создание колод и карточек.
- Повторение карточек по системе интервальных повторений.
- Оценка ответа: `Again`, `Hard`, `Good`, `Easy`.
- Просмотр прогресса по колодам.
- Добавление карточек вручную.
- Генерация карточек через AI в будущих версиях.
- Статистика удержания, streak и активности.

## MVP

Первая версия должна быть локальной и автономной:

- onboarding;
- главный экран с карточками на повторение;
- список колод;
- экран колоды;
- создание карточки;
- сессия повторения;
- базовый алгоритм интервальных повторений;
- mock data или локальное хранилище на раннем этапе;
- unit-тесты для SRS/Leitner-логики.

Подробный порядок разработки и критерии готовности этапов описаны в [roadmap](docs/roadmap.md).

Базовые дизайн-токены из прототипа:

- primary green: `#2D8653`;
- accent coral: `#E5614A`;
- info blue: `#4585B8`;

Планируемые шрифты:

## Технологический стек

- `bloc` и `flutter_bloc` — управление состоянием через Bloc/Cubit;
- `drift` — локальная реляционная база данных;
- `freezed` и `json_serializable` — модели и кодогенерация;
- `sheety_localization` — генерация локализации;
- `talker` и `talker_flutter` — логи, ошибки и диагностический интерфейс;
- `theme_tailor` — генерация типизированных `ThemeExtension`;
- `flutter_svg` — отображение SVG;
- `fl_chart` — графики статистики;
- `spider` — генерация типобезопасных ссылок на ассеты.

`talker_dio_logger` будет добавлен только вместе с `dio`, когда появится сетевой слой. Основные данные приложения не должны зависеть от логгера.

Для растровых изображений предпочтителен формат WebP. PNG используется только при обоснованной необходимости, а масштабируемые иконки и простая векторная графика хранятся в SVG.

## Архитектурные принципы

Проект развивается по принципам SOLID и прагматичной Clean Architecture:

- **Single Responsibility**: экран, виджет, сервис, репозиторий и use case отвечают за одну конкретную задачу.
- **Open/Closed**: новую бизнес-логику добавляем через новые реализации и композицию, а не через разрастание условий в существующих классах.
- **Liskov Substitution**: реализации интерфейсов должны быть взаимозаменяемыми без изменения поведения вызывающего кода.
- **Interface Segregation**: не создаем слишком широкие контракты; интерфейс должен описывать только нужные операции.
- **Dependency Inversion**: высокоуровневая логика зависит от абстракций, а не от конкретных API, базы данных или SDK.

Clean Architecture используется как способ отделить бизнес-правила от Flutter, хранилища и внешних сервисов, а не как требование создавать одинаковый набор слоев для каждой фичи.

Для фич со значимой бизнес-логикой используются слои:

- `domain` — сущности, value objects, контракты и чистые алгоритмы;
- `application` — use cases и координация сценариев;
- `data` — DTO, источники данных и реализации репозиториев;
- `presentation` — Bloc/Cubit, экраны и виджеты.

Направление зависимостей:

```text
presentation -> application -> domain
data -----------------------> domain
```

При этом:

- `domain` не зависит от Flutter, Bloc, базы данных, API и сторонних SDK;
- Bloc/Cubit управляет состоянием представления, но не содержит SRS-алгоритмы и SQL-запросы;
- внешние технологии подключаются через локальные интерфейсы и адаптеры;
- простые UI-фичи не обязаны иметь `domain`, `application` и `data`;
- use case создается только для самостоятельного бизнес-сценария, а не для каждого нажатия кнопки или чтения поля.

Цель архитектуры — обеспечить разделение ответственности, тестируемость и заменяемость инфраструктуры без лишнего boilerplate.

## Предлагаемая структура

```text
lib/
  app/
    dependencies/
    logging/
    resources/
    startup/
    navigation/
    theme/
  core/
    errors/
    utils/
  features/
    onboarding/
      presentation/
    home/
      presentation/
    decks/
      domain/
      application/
      data/
      presentation/
    cards/
    review/
      domain/
      application/
      data/
      presentation/
    stats/
    profile/
  shared/
    widgets/
```

Структура может меняться по мере развития проекта. Слои добавляются только при наличии соответствующей ответственности: например, простому onboarding может быть достаточно `presentation`, а SRS-фиче нужны независимые domain-алгоритмы и тесты.

Точка входа `main.dart` передает запуск в `startup()`. Там настраиваются глобальная обработка ошибок, Talker, Bloc observer и зависимости приложения. Зависимости создаются явно и передаются через `AppDependenciesScope`, без service locator.

## Навигация

На старте проект использует стандартный императивный `Navigator` с `MaterialPageRoute`. Это покрывает линейные сценарии MVP без дополнительного routing-слоя:

```text
Onboarding -> Home
Home -> Deck
Home -> Review -> Result
Deck -> Add card
```

Собственный router или сторонний пакет не добавляется заранее. Решение пересматривается, когда появятся конкретные требования к deep links, web URL, state restoration, сложным guards или нескольким независимым стекам навигации.

## Генерация кода

Основная команда:

```bash
dart run build_runner build
```

Spider устанавливается глобально и не добавляется в зависимости проекта:

```bash
dart pub global activate spider
spider build
```

Его конфигурация хранится в `spider.yaml`, а сгенерированные ссылки на ассеты — в `lib/app/resources`.

## Проверки

Базовая проверка проекта:

```bash
flutter analyze
flutter test
```

Дополнительная локальная проверка через установленный DCM:

```bash
dcm analyze lib test
```

DCM пока не является обязательной частью публичного CI, поскольку требует отдельной установки и CI-лицензии.

Для изменений в бизнес-логике повторений обязательно добавлять или обновлять unit-тесты.
